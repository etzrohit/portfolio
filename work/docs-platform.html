Context

Why the docs mattered

Facets asks people to work in a new way: describe what they need, and the platform builds the infrastructure. The docs are how customers learn to do that on their own when we can't put someone on a call.

  • The product ships every week. The docs, written by hand, could never keep up.
  • People hit a page that no longer matched their screen, and stopped trusting the docs entirely.
  • So they came back to us with the same questions, call after call, across almost every account.
  • It hurt most during trials, where a customer has to prove they can run Facets before time runs out.
Users & strategy

Who the docs had to serve

Before changing anything, I looked at who actually relies on the docs and what each group needs.

Who What they need Why it matters to the business
01 Trial engineers Setup steps they can follow without a call, before the trial runs out Decides whether a trial converts
02 The 2nd and 3rd engineer on a live account A reference they can trust, since no call is booked for them Expansion and retention inside paying accounts
03 Praxis adopters Docs that keep up with the fastest-moving product Praxis is the growth bet; weak docs stall it
04 Support, CS, and engineers Docs that are cheap to update, and a source of truth to point to Every stale page turns into a support call
What I focused on

Customers can only get value from Facets if they can learn it on their own, and the docs are how they do that. So I aimed at two problems:

  • Make answers easy to find, so more people finish setup on their own. That helps trials turn into paying customers.
  • Keep the docs current without manual upkeep, so we spend less time on support and free up the team.
Research

How I found the problem

The docs looked fine on the surface. The real problem was that keeping them accurate was all manual, so they always fell behind.

  • Checked every page against the live product, and asked the engineers and PMs who own each area which topics drift the most.
  • Traced how a doc actually gets written. The bottleneck was the workflow, not the writers.
  • Asked who writes docs and who reads them. Both pointed at the same fix: make docs cheap to keep current.
Problem

What I found

The docs were not behind because anyone was careless. They were behind because the product ships every week, and no one can hand-write docs fast enough to keep up. Every idea on the table was really just "write faster."

  • Docs drifted out of date faster than anyone could fix them. A reader had no way to tell which pages were current, and one wrong page cost trust in all of them.
  • Writing a single doc was too much work. Docs, images, and code all lived in different places, so small updates never happened.
  • The reading experience was dated: flat navigation, weak search, and tiny images with no way to zoom.
  • Praxis, the most-asked-about product, was barely covered, even though it shipped the fastest and was hardest to keep current by hand.
Prioritization

Why I built a whole portal

I did not set out to rebuild the docs. I set out to automate doc writing. Chasing that is what led to the portal.

How the decision actually happened

To automate the docs, the agent has to open a pull request every time it updates a page. Our docs lived on readme.com, where automated PRs were complicated and needed an extra premium plan.

So I made the call: move the docs into our own codebase and build a native portal. That turned out to be the easy part, and it unlocked the automation, the structure, and the reading experience all at once.

Value

High

Docs are the fallback for a self-serve product. When they fail, people come back to calls and trials stall right where they need to prove they can run it themselves.

Effort

Lower than it looked

Building a native portal in our own repo was simpler than bending readme.com into automated publishing, and it dropped the premium cost. A big payoff at a small price.

How I chose them

Page count proves nothing. A docs site is only doing its job on two counts: are the docs current with the product, and can a reader actually find the answer. Everything I built points at those two, not at how much content exists.

Decision

The two calls I made

Call 1

Rebuild the experience, not just move the pages

Moving the old docs as-is would carry over the flat nav, weak search, and tiny images. So I restructured the whole information architecture and built real authoring components: click-to-zoom images, interactive demos, and section cards.

Call 2

Stop writing the docs by hand

If we can never hand-write docs fast enough, a better writer or stricter process won't fix it. The docs have to be written from the same place the product comes from: the code.

So I built Doxy

Doxy reads the actual Facets and Praxis codebases: how a release behaves right now, what the debugger checks when something breaks, and writes the docs from that, not from someone's memory of how it used to work. It never ships on its own:

  • Grounded in real code, so it cannot invent a button label, a permission, or a setting that does not exist.
  • Two human gates, so nothing is written until I approve the plan, and nothing is published until I have seen it previewed next to the live site.

Now a doc can be drafted from the same change that shipped it, reviewed, and published the same day, instead of waiting for someone to find time they never had.

How the agent works

Doxy, step by step

One orchestrating skill plus seven subagents, about 2,300 lines of instructions and no application code. Every run follows the same path, and it stops for me at two hard gates: what to write, and whether to publish.

Reflection

What I'd take forward

Worked well

  • Fixing how docs read and how they get written at the same time. A nicer site that is still painful to update would have drifted again.
  • Grounding the agent in the code. That is the difference between docs you can trust and an agent that makes things up.

Would do differently

  • Build the agent earlier. I hand-wrote pages that Doxy could have drafted, then reworked them once it existed.

The lesson

  • The hard part was not writing the docs. It was realizing that no one could ever write them fast enough by hand. Once I saw that, the fix was clear: let the code write the docs, instead of a person.

Context

Why the docs mattered

Facets asks people to work in a new way. Instead of filing a ticket and waiting, they describe what they need and the platform builds it. That is a big change, and it raises fair questions at every step:

  • Is a project the same as an environment?
  • Why is my release waiting instead of running?
  • Can I trust the AI debugger, or should I check it myself?

We could only answer those on a call, one or two people per account. We taught whoever joined first and hoped they passed it on. That never worked: the next engineer hit the same wall, with no call booked for them.

The docs were meant to answer these instead. But they were always behind the product, so people landed on a page that no longer matched their screen, and stopped trusting the docs at all.

So they came back to us: almost every customer, not just one. The same questions, call after call. It cost the team real time, and it hurt most during trials, where a customer has to prove they can run Facets before time runs out.

Research

How I found the problem

Checked every docs page against the live product

I went page by page and compared each one to what the product actually did, and sat with the engineers and PMs who own each area to learn which topics drift the most. Whole sections were out of date, screenshots showed old screens, and Praxis had almost no docs even though it shipped the fastest.

Traced how a doc actually gets written

Writing a page meant leaving the codebase, editing in a hosted tool, and hosting images somewhere else again. Every step added friction, and friction is why updates kept slipping. The bottleneck was the workflow, not the writers.

Asked who writes docs, and who reads them

Engineers had the knowledge but treated docs as an afterthought, because it sat outside their flow. Readers had learned not to trust the docs, so they asked in Slack instead. Both point at the same fix: make docs cheap to keep current.

Problem

The four structural problems

1. Docs drifted out of date faster than anyone could fix

The product shipped every week and the docs did not. Pages described screens that no longer existed, and a reader could not tell which pages were current. Once someone gets burned by one wrong page, they stop trusting all of them and go back to asking a person.

2. Writing a doc was too much work

Docs lived away from the code, images lived somewhere else again, and there was no fast way to preview a change before publishing. The cost of every small update was high, so small updates never happened.

3. The reading experience was dated

Navigation was flat and hard to scan, search was weak, and images were tiny with no way to zoom. Interactive product demos broke their own sizing. Finding an answer took longer than it should.

4. The newest, most-asked-about product was barely covered

Praxis, the AI layer, was what customers asked about most, and it had almost no docs. It was also the hardest to keep current by hand, because it changed the fastest.

Information architecture

The IA decision in depth

Three problems with the old structure, all structural, not cosmetic

  • No journey. A brand-new trial engineer and a two-year admin saw the exact same flat list. Nothing told either one where to begin or what came next.
  • Named for the product, not the person. "Environments", "RBAC", "Modules" are how we talk internally. A customer arrives thinking "how do I set this up", "how do I get my app in", "how do I control who can touch production". The labels did not match that language.
  • The same page lived in several places. Copies drifted, so two pages that should have said the same thing said different things, and the reader could not tell which was right.

The options I weighed

Group by Why not / why
A The product's internal modules This is the structure we already had. It is the org chart, not the user's head. Rejected.
B The codebase layout on disk Convenient for us, meaningless to a reader. It optimizes for the writer, not the user. Rejected.
C The user's job, in adoption order Matches how a team actually moves through the platform: learn it, set it up, bring an app, ship, operate, govern. Chosen.

The evidence that pointed at structure, not content

  • The page-by-page audit showed good content was unfindable because it was filed under a product noun a reader would never think to click.
  • The onboarding calls were a live card sort: the recurring "is a project the same as an environment" is a findability failure, not a content one.
  • Support and Slack threads showed people asking for answers that already existed on a page they never found.

Three sentences that framed every decision

  • Users struggled to figure out where to start and what to do next.
  • The old structure made that harder because it was a flat list named after the product's internal parts, with no order and duplicated pages.
  • The new IA fixed it by grouping every page under the job a user is doing, in the sequence they do it, and speaking the user's language in the labels.

Why Praxis and Reference are split out

  • Praxis (Agents) is the growth bet and the fastest-moving product. It needed its own home so it stops competing with the everyday guides for attention, and has room to grow without pushing the journey down the page.
  • Reference (CLI, API, Web Components, Integrations, Support, Changelog) is a different intent entirely. You do not read reference, you look something up. Separating lookup from learning keeps the daily journey clean and short.

How the navigation is actually built

  • Two-layer model. Sidebar order lives in small meta.json files; the pages live in a content warehouse. Change the order without touching content, reuse a page without copying it.
  • Verb labels over stable slugs. The reader sees "Setup Platform" while the page still lives at /docs/configuration, so I could reframe every label without breaking a single link.
  • Separators and icons. Grouping headers (PRAXIS, REFERENCE) and a per-section icon make a long sidebar scannable at a glance.
  • The silent nav trap. A section can list its pages one by one, or point at a nested folder as a single entry. Get it wrong and the page still builds but never shows in the sidebar. This exact bug bit real content during the restructure, so the rule is now written down and checked on every change, which is also why the docs agent handles it safely.
  • Isolated from marketing. Docs run their own layout, sidebar, search, and theme, so a docs change can never break the marketing site, and the other way around.
  • Nothing broke on the move. 178 old readme.com URLs redirect to their new homes, so inbound links from Google and bookmarks still land.

What the restructure changed for the reader

The point of all of it was one number: can a reader find their answer. Task success finding an answer went from 64% to 90%, and the new structure is what made the automated writer possible, because there is now exactly one place each fact lives.

How the agent works

Doxy, in depth

Why seven subagents instead of one prompt

One orchestrator handles intent, cloning, the plan gate, and the preview gate. The code-reading layer stays split by product on purpose: Facets and Praxis live in different repos, in different languages, with different security rules. A single mega-prompt would either bloat every run with irrelevant branches or blur the real distinctions.

graph TD

    R0 -->|Facets| FR["facets-reader"]

    R0 -->|Both| FR

One orchestrator, product-specific readers. Seven subagents in total, plus a shared changelog writer.

The design decisions behind it

  • Generate from code, not from PRs or a human template. The code is the only source that cannot lie about what shipped. This is what kills drift, and it is how the agent caught a documented step that no longer existed.
  • Plan, then write, with a human gate. Nothing is written until the per-page plan is shown and approved.
  • Preview versus live, then PR. Nothing is pushed until the local preview is compared against the live page and I say "raise pr."
  • Disposable clones only. Every run clones fresh into a temp folder and is discarded. Nothing touches a real working tree until an explicit PR.
  • Always ask, never infer. The product (Facets, Praxis, or Both) is always confirmed, never guessed from a keyword.
  • Security filter before any write. Readers redact credentials, internal URLs, internal DB and collection names, and internal identifiers. The rule is "describe the capability without the sensitive detail," not omit the capability.

How it got here

  • Started as two skills, one per product. The orchestration was duplicated almost byte for byte, so I merged it into one /doxy and kept only the code-reading layer split.
  • Added the grounded-in-code rule and the security filter after the writer could over-claim or risk leaking internals.
  • Added the local-preview-versus-live gate, because direct publish was too risky.
  • Encoded the silent-nav-trap detection after it bit real content during the restructure.
  • Shipped as a team plugin, with two live features written end to end and merged.
