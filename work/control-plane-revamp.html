Context

Where things stood

This wasn't the first attempt. Facets had already tried to redesign inside its Angular app. Customers pushed back hard enough that two UIs had to run side by side for seven to eight months, and keeping both alive took roughly a quarter of the two-person frontend team's capacity.

A re-platform was already underway. Separately, Facets was moving its frontend from Angular to React for engineering reasons: framework limits, hiring, and long-term maintainability. Every screen was going to be rebuilt regardless.

That made this the moment. A straight lift and shift would have carried the same interaction problems into a new framework, and the team would have paid to fix them a second time later. Since the screens were being rebuilt anyway, fixing the experience added very little extra cost.

The problem

Everyday tasks took too many clicks, the navigation was confusing, and the UI was slow enough to get in the way.

Who felt it

Developers

Shipping changes.

DevOps and platform engineers

Keeping environments healthy.

Not "users" in the abstract. They ran the same screens dozens of times a day, so every extra click and every "where is this again?" added up quickly.

In interviews and sessions, the same three words kept coming up: confusing, slow, too many steps.

Cost of leaving it alone. A control plane that is slower than a script pushes people back to scripts and runbooks, the exact thing Facets exists to replace. On infrastructure that 30+ companies run in production, that is a retention risk, not a cosmetic complaint.

Prioritization

Why now

We scored this on value versus effort, and it stood out on both.

Value

High

A daily tax on 30+ paying customers' production work, the kind of friction that pushes people back to scripts.

Effort

Unusually low

The React re-platform was already touching every screen, so fixing the interaction model came at little extra cost, not a separate project.

Why now specifically. The re-platform was a one-time window to fix the information architecture while every screen was being rebuilt anyway, and the seven-to-eight-month dual-UI period was still fresh for everyone.

The upside. Removing per-task friction for around 40 daily operators, later measured at about 150 engineer-hours saved every month, plus lower churn risk.

Goals

What success looked like, set before we built

The user goal underneath every target was simple: let someone finish a routine task, releasing, rolling out, or overriding, without hunting for where it lives. Four targets, each with a metric and a baseline, were agreed before any screen changed.

01

Support tickets asking "who owns this" or "where does this live." Baseline ~19 a month, goal to cut it by more than half.
02
One navigation and settings system
The separate, one-off drawer patterns across CI/CD, Release, and Role. From 3 patterns down to 1.
03
Fewer clicks to finish the job
Clicks to complete the flagship daily task. From 14 down to under half.
04
Faster to finish a task
A combined time-and-steps score across the five core flows (trigger a release, launch an environment, configure and override a resource, update an image, debug a failed release), set to 100 before the redesign, aiming for a clear reduction.
How we picked what to measure

Usage is the wrong metric here: customers have to release, launch, and configure regardless of UI quality, so it would have flattered us without proving anything. Instead we split screens by choice. For the unavoidable ones (Release, Launch, config) we measured friction: time, clicks, drop-off. For the optional ones (dashboards, cost) we measured whether people came back and acted, because there a visit is a real vote.

The problems we found

01

Org, project, and environment work were tangled together

There was no clear line between them. The same navigation showed up everywhere, even deep inside a project.

02

Moving between pages was its own task

Jumping to the release page, the environment page, or switching between project-level and environment-level resource configuration each meant a completely different flow.

03

Small tasks took too many clicks

Releasing a change, rolling out an update, or overriding a resource each took more steps than they should have.

04

Releasing ended on a blind commit

The final step shipped the release with no preview of what would actually change.

05

And a long tail

Settings drawers each behaved differently, there was no dark mode, and two icon sets were running side by side.

Some of this started as literal marker-on-laptop scribbles: mapping environment config rules, dynamic launch overrides, and the color token and icon migration underneath the redesign.

The key decision

The big bet: get navigation right first

Every other screen sat on top of the navigation. Get it wrong and we would have had to reshuffle every page later, so we settled it first. We finished the navigation and shipped it in version one, along with all the org-level settings, gathered feedback, folded it in, then built the rest as version two.

Explored

Environment-first

A top bar switched between the blueprint (project level) and the environments, and that choice rewrote the side nav beneath it.

Why it broke down: the two levels never lined up. Picking "Blueprint" surfaced project-only actions like Pipeline and Cost that do not exist for an environment, so the side nav kept reshuffling.
Shipped

Context-switch

Open a project and everything starts at the project level with one steady side nav. To go deeper, you drill into an environment, or jump straight there from Project Overview.

Why it won: it matches how people already picture the hierarchy, so there is nothing to memorize, and the side nav stays consistent. You follow one path: project, then environment, then resource.
A design-system call

The migration had to move fast, so we adopted Ant Design instead of building a system from scratch. We built on top of it and made custom components only where it fell short, which kept our time on the flows and the hard screens.

Solution

The screens, and what moved

One navigation model, dashboards that answer "what is the state of my thing" at a glance, and a shipping flow that shows what is about to change before you commit. Three principles held it together: put the action where the task already is, reuse one pattern instead of a bespoke layout per screen, and show the consequences before the commit.

1/5 Information architecture

Two navigations, not one shared menu

Outside a project the sidebar shows org-level work; step inside and it is replaced entirely by project-scoped navigation. The two levels no longer share a menu, and the hierarchy is explicit: project, environment, resource.

Launch

How we shipped it

This runs production that other companies depend on, so there was no big-bang cutover. We shipped in two versions.

Version 1

The foundation

Navigation, plus all org-level settings and configuration. It went out first, so the highest-dependency work was settled and customers had something real to react to before we built on top of it.

Version 2

The full redesign

The project and environment dashboards, the shipping flow, secrets and variables, and dark mode. Built after V1 feedback was in.

How V2 reached customers

Gated per customer org behind a feature flag, so any org could roll back without a redeploy. It went to a pilot group first, then to everyone, with the old UI kept as a fallback for a short bake period before both legacy UIs were retired. Within four weeks, every customer had accepted and adopted the new UI, with 100% feature adoption.

Reflection

What I learned

🤖 Worked, and I'd do it again

There was no existing workflow for a designer shipping straight into production.

  • Designing directly in code. Figma directions went into the live React codebase through Claude Code, as PRs reviewed like any contributor's, with no separate design-to-engineering handoff.
  • One consistent pattern over several "good enough" ones, even when it meant redoing recent work.

⏱️ What I'd do differently

Lock the foundation first.

  • Two dashboards shipped before the dark-mode tokens were final, which forced avoidable rework a few weeks later. Freezing the tokens first would have prevented it.

🧭 What I had to navigate

Every change was reviewed by the engineers building the same product.

  • Working inside someone else's codebase, I scoped changes to be reviewable under the existing conventions and chose carefully which refactors were worth pushing for.
