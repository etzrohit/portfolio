Work / Loyalty Promotion

Automating the creation and management of Loyalty Promotion.

This case study is about adding new features and improving the user experience on top of the existing UI, using the current design components. It's not a complete UI redesign—the main goal was to enhance functionality and solve user problems without changing the design that's already in place.

Background context

What is a loyalty program?

A loyalty program is a marketing strategy that rewards customers for returning to a brand, encouraging them to keep shopping and make more purchases. Customers earn coupons and points based on their engagement and activities, like transactions, helping brands influence behavior, build loyalty, and boost repeat business.

What is a promotion?

Promotions are short-term loyalty programs that offer extra rewards on top of the basic loyalty program. You might have seen, that during a festival season or to promote a specific product, brands can boost sales by giving customers additional points or coupons.

A most basic example of promotion can be:
- If a customer buys something worth more than $500, they get 50 points.
- In flipkart if you buy something you get 4 supercoins for each 100 rupess you spend. Similarly in case of credit cards you get rewards based on the purchase you do.

To make these possible brands write these these above conditions in the form of rules(codes) to create the promotion.

What is CPG, FMCD, and FMEG?

Consumer packaged goods (CPGs) refer to the space within an industry that features goods that consumers use in everyday life.

FMCD & FMEG Brands are a subset of CPG.
  • FMCD (Fast-moving consumer durables) - FMCDs are durables like TVs, refrigerators, washing machines, microwave ovens, etc which have a longer shelf life in years.
  • FMEG (Fast-moving electrical goods) - A subcategory of the above includes household multi-use products like bulbs, LED, etc.

Problem Statement

A CPG company has to write 1000s lines of rules to create a promotion on top of it, management (editing, testing, and distribution) of these rules manually is both time-consuming and error-prone.

Introduction

This feature allows brands to automate promotion creation, and management in bulk bringing down writing 1000s of rules to using an upload-based system. With the help of this feature, brands can now deliver the right promotion to the right distributors, retailers, and consumers efficiently and faster.

  • When creating a promotion, users will have two options:
    • Create a manual workflow (existing flow)
    • Upload rules in bulk (new flow)
  • Users will select a template.
  • They’ll upload a CSV file containing the promotion rules.
  • Once uploaded, the system will validate the CSV against the selected template and flag any errors.
  • Users can then simulate the rule using an existing test transaction.
  • After this, the user can submit the promotion for approval.
  • The approver can either accept or reject the promotion, adding any necessary notes.

Core users

  • Brand Managers: The point of contact between the brand and Capillary. They gather all requirements from the brand and manage the brand’s loyalty program.
  • Loyalty Program Administrators: The configurators who handle all the setup and configuration related to the brand's loyalty program.

Research

Understanding the current behaviors of brand managers and loyalty program administrators in gathering requirements from the brands.

It began with multiple calls with various brand managers and their program administrators to understand their current behaviors and the jobs they need to accomplish (JTBD).
  • How do they gather requirements for different promotions?
  • Once the requirements are received, how do they create promotions and track all the promotions created? With so many promotions, how do they manage their distribution?
  • How do they test the created promotions?
  • After creating a promotion, what changes are typically made, and how frequently?
  • How do they manage promotions that have already been created?

Understanding the Problem

How does the current system work?

Right now, the promotion creation page uses a workflow where rules are manually written for each promotion. These rules are structured in sets (Parent), where each set can have multiple conditions (Child), and each condition can contain several rules.

These conditions follow an if/else logic. For example: If condition 1 is true, then issue points; otherwise, check condition 2.

Example Promotion

Let’s take an example of promotion to understand how promotion is created and what are the main problems.

Brands provide these sheets to the brand manager which contain details of the promotion like start date, end date, SKU, Product category, benefits to be given, etc.

In the above sheet, there are three highlighted columns in green containing promotion details like product code, type and no of points to be awarded.

Simple explaination of what’s written in that sheet:
- if a customer buys product 1 (whose product code is FPA837405000041) from shoes category then award that customer with 225 points.
- if a customer buys product 2 (whose product code is FFPA843800200600) from watch category then award that customer with 5 points.

These are the data that will go in the rules and act as the spine of the promotion. These data types can be different for different brands.

Above we have 3 Product types, and 100 Products in each type

  • For shoes, the total rules will be 100 (10 rules for P1 and 90 for P4), similarly 100 for watches and glasses as well
  • For this simple configuration, the total number of rules becomes 300. This is the case when there are only two variables such as Type and Product code.

This is how the rules are written when there are different variables:

If the product user buys matches any of the product codes mentioned above in condition 1 they will get the points, if not the system will check in conditions 3, 4, and so on.

This example covers rules with only one variable (product code) if more variables like product type, shop location etc come in picture the rules will become more nested.

Key Problems

Some of the major problems that users are facing is:

  1. Scalability Issues: The parent-child and if-else logic become cumbersome and inefficient for large-scale programs. The system struggles with managing more than 100 promotions effectively.
  2. Nested Rules: Rules can be nested, creating a complex hierarchy that's tough to debug. This can make the structure resemble code, which is challenging for non-technical users to understand and manage.
  3. Accuracy and Trust: The existing UX does little to assist users in configuring promotions, leading to potential errors and a lack of trust in the system.
  4. Hard to Edit: The client provides a points table based on customer, SKU, and transaction attributes that change monthly or quarterly. The current loyalty rule setup complicates finding the exact rule across multiple sets and conditions for accurate updates.
  5. Not Suitable for Brands with Multiple Loyalty Programs: Clients often need to award points to retailers, dealers, and distributors within a single transaction. Common solutions create separate organizations, leading to complicated data mapping.

Solution

Based on the insights, we decided to utilize the sheet provided by brands, because we were able to identify a pattern in the promotion.

For example: Company X promotions are driven by the following factors:

  • Product (SKU/category, etc.): There can be up to 100 different products.
  • Location (Area/Zone, etc.): This leads to hundreds of combinations, with 4-5 zones, 10-20 regions per zone, and further divided into areas.

This structure can result in thousands of rules (e.g., 100 products x 100 locations = 10,000 rules)

We created few templates for different types of promotions which can cover around 80% of the brands use-case. The idea was that the user will upload the sheet provided by brand and the system will map this data with the selected template and create the promotion without user writing any rule.

How does a template work and eliminate the rule-writing process:

Here's what happens when you select a template and upload a file:

  • Each template is mapped with some function.
  • For eg: A template X is mapped with Fn(a,b) and Fn(x,y)
  • The values of a,b, and x,y are coming from the columns of the file uploaded by the user.
  • Each column, like category or line item, uses an AND condition (they all need to match) and is linked with a function in the backend.
  • Each row is a rule and operates on OR logic (if any row fits, the rule applies).
  • There can be multiple templates for a single brand covering different types of promotions. The user will have to select the template based on the type of promotion they want to create.
  • In this example, for Row 1, if all the columns match, the system will give 100 points. The same applies to Row 2. If everything matches, 100 points will be provided.

Template creation process

I did a ton of research and exploration for the template creation, but I can’t share all the details. What I can share is how I simplified the existing workflow and turned it into a user-friendly template.

I started by looking at the different promotion sheets from the brand and how they were currently implementing these promotions in the workflow.

For example, take the brand “ABC.” They have two loyalty programs: one for customers(Cx) and another for distributors(Dx). Most promotions focus on a few KPIs, like SKU, transaction value, and other custom KPIs.

  • Here’s a simple case: they might say, “Give XYZ points on these 100 SKUs.” To do this, the configurators have to repeat the same rules over and over using “OR” logic, only changing the SKU values that too for two times for loyalty promotion Cx and Dx.

This gets more complicated with multiple sets and conditions. When they have to handle different benefits or categories using “AND/OR” logic.

The process for creating the template involves:

  • Removing any irrelevant data for promotion creation.
  • Filtering and organizing the data needed for the UI and the template (which data we will ask in UI and which data we will fetch from the uploaded file).
  • Analyzing the promotion variations for each brand to determine how many templates are needed to cover 80-90% of use cases.
  • A lot of edge cases and other factors came into consideration which I can’t disclose.

Designs: First-time promotion creation using the bulk upload flow

We explored numerous enhancements and sub-features but eventually narrowed it down to just the essential ones for phase 1.

File upload flow

  1. Promotion rule selection is added to the existing flow:
    • Manual Rule: This is the old flow where configurators create promotions by writing rules in the workflow.
    • Upload Rules: This option allows users to skip the old workflow and directly upload a CSV file to create the promotion.
  2. If the user selects the “Upload Rule” option, the system will prompt them to choose a template.
  1. Once the template is selected, a preview of the template is displayed.
    • The preview includes the expression name, expression definition, and its type.
  1. After selecting the template, the user will upload the CSV file.
    • A preview of the template mapping with the uploaded file will be shown after the upload.

      ↗️ Background context on how template and file mapping work.

    • The preview includes the mapping of column headers from the uploaded file to the column headers in the template, the column header types, and a preview of the information.
  1. Once the file is uploaded, the user will need to validate it.
    • The validation checks for any errors in the uploaded file. If errors are found, the system will provide a new file highlighting the erroneous rows.

Define Action

  1. Define Action basically means determining what will happen once the rules are satisfied. Actions represent the benefits, which can take the form of points, coupons, or vouchers. For phase 1, we are focusing solely on points.

Why is it necessary to define an action when the number of points to be given to the user is already in the uploaded file?

  • A point action includes various metadata, such as:
    • Earn and expiry conditions for those points.
    • Whether the points are percentage-based (e.g., 10% of the transaction value) or fixed value (e.g., 100 or 200 points).
  • The uploaded file only includes the point value, but it doesn't contain the other metadata.
  • For a promotion, all points awarded share the same metadata.
  • Therefore, the user only needs to specify the number of points in the file, while the other metadata will be provided through the action details.

Simulation of Promotion

  1. Once the promotion configuration is complete, the user can test it by passing sample transactions.
    • After selecting a customer, all transactions for that customer will be displayed.
    • Since many promotions apply only when the transaction amount exceeds a certain value, showing the transaction amount upfront is crucial. This helps the user decide whether to test the promotion with that transaction or not.
  1. Based on the selected customer and transaction number, the details of that specific transaction will be displayed.
    • Users can adjust the amount, quantity, and discount for each transaction because many promotions apply only when the line item quantity exceeds a certain threshold. This allows users the flexibility to customize the bill as needed for testing.
  1. After testing the promotion, the points details will be visible for each line item.
    • Since multiple promotions can apply to a single transaction, users will see a breakdown of the allotted points, showing which promotions are applicable and how the points are distributed among each promotion.
  1. Compare with Original Earn: If the user wants to see how the points change across all applicable promotions, they can turn on this toggle.
    • In the original earn view, users can see that the tested promotion “spin-the-wheel” didn’t award any points because it isn’t live yet. Initially, the “Christmas offer” was giving 150 points, but now it’s only giving 100 points.
  1. Once the simulation is complete, the user can send the promotion for approval.
    • The approver will review the transactions simulated for this promotion. If they believe any other transactions should be tested before going live, they can test those and then make the promotion live.

Results

  • This was the MVP of a larger project where we aimed to test its scalability with a few brands.
  • The brands loved this new approach because it saves configurators a lot of time. Editing anything post-creation is straightforward; they just need to edit the file and re-upload it instead of sifting through sets and conditions to find the right rule.
  • The simulation flow effectively reduces errors. Previously, promotions went live in the testing environment, and configurators would have to correct errors afterward, which was time-consuming and less effective.
  • Every stakeholder understands the CSV file and how these rules work, which helps drive higher participation and loyalty among all parties involved.
  • The project's extension includes many new features, like creating multiple promotions by uploading data and developing complex promotions with milestones and streaks, which I can't discuss in detail here.
